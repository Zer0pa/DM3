# Phase 01.2.3.4.1.1 Context

## Locked Decisions

- This is a capability-discovery reset on the RM10-primary branch.
- Delete drift from the active surface; do not merely route around it.
- Make exactly one sealed rescue pack before destructive cleanup.
- RM10 is primary. Mac is support only for contrast, debugging, and backup.
- Determinism, canonical hash agreement, and default-validator success are
  secondary telemetry in this phase, not the front-door stop for exploratory
  batteries.
- The sovereign acceptance gate for authority does not move: exploratory signal
  discovered here does not become a governed pass narrative by itself.
- Do not preserve stale validators, shadow workspaces, duplicate wrappers, old
  runners, reused outputs, or hidden fallback paths in the live execution path.
- Do not narrate capability from callability, runtime length, GPU heat, or
  utilization.
- Do not widen into ontology, architecture mythology, or new scientific
  doctrine while the discovery question is still local and experimental.

## Mission Order

1. Seal one rescue pack.
2. Reduce the active surface to one live RM10 workspace root, one live launcher
   path per tested lane, one Mac support workspace, and one live output root.
3. Prove only that the cleaned lane still runs.
4. Run a capability smoke lattice on the cleaned RM10 lane.
5. If signal exists, map its properties before returning to handoff doctrine.
6. Test same-family handoff truth only as a property-preservation question.
7. Contrast lanes only when the alternative lane is callable and interpretable.
8. Attack the apparent effect adversarially before any recommendation.

## Primary Question

Can the RM10-hosted system do anything real, nontrivial, and characterizable at
all once drift is stripped away?

## Subsidiary Questions

1. Does the cleaned system produce any measurable response to controlled
   perturbation?
2. Are there response families rather than random noise?
3. Are there thresholds, mode flips, persistence, hysteresis, or recovery
   effects?
4. Do boundaries and handoffs carry disproportionate signal?
5. Does lane choice alter behavior or only runtime?
6. Does the apparent signal survive hostile falsification?

## Scientific Stance

- Treat the object experimentally rather than doctrinally.
- Be open to: a real effect, a weak effect, a boundary-mediated effect, a
  lane-sensitive effect, a narrow-precondition effect, or no effect at all.
- A fast honest negative is better than validator theatre around an empty
  object.

## Working Hypotheses

- `H1`: the cleaned RM10 single-lane surface shows some structured response to
  controlled perturbation.
- `H2`: if signal exists, it is boundary-sensitive and disproportionately lives
  at interfaces, packet boundaries, selectors, or constrained transitions.
- `H3`: some properties may stabilize in kind before they stabilize bitwise.
- `H4`: compute lane may alter behavior, not merely speed.
- `H5`: the apparent effect may collapse completely once drift is removed.

## Signal Standard

Signal in this phase means one or more of:

- a consistent response class
- a non-random sensitivity map
- a threshold or phase transition
- hysteresis
- persistence across short reruns
- a boundary-transfer effect
- a lane-dependent behavioral change
- a measurable thermal or cost trade paired with behavioral change
- a clean falsification of a supposed effect

Running, heating, or emitting receipts alone is not signal.

## Minimal Discovery Observable Contract

Each serious battery must name:

- one anchor observable
- one drift observable
- one abstain rule

Admissible anchors include response magnitude, convergence class,
boundary-window coherence, packet summary fields, energy-like descent, or phase
classification if the surface really exposes them.

Inadmissible anchors include visual plausibility, filenames, runtime duration
alone, GPU utilization alone, or prose similarity.

## Battery Sequence

- Battery A: drift purge and clean-room bring-up
- Battery B: capability smoke lattice
- Battery C: property mapping
- Battery D: same-family handoff truth
- Battery E: lane contrast
- Battery F: adversarial falsification

Battery D runs only if Batteries B and C justify it.
Battery E runs only if the lane is callable and interpretable.

## Logging Discipline

Every serious run must preserve:

- run ID
- battery ID
- lane
- exact command
- cwd
- output path
- before and after battery state
- before and after thermal state
- short observable summary

Optional telemetry such as hashes, validator output, and extended process data
is allowed but does not become the front-door gate in this phase.

## Deletion And Quarantine Rules

- If an artifact silently redirects execution, duplicates the active binary,
  revives an old validator surface, reuses old outputs as current truth, or
  keeps fallback ambiguity alive, it must be deleted from the active path or
  quarantined outside it.
- The sealed rescue pack is the only sanctioned safety blanket.
- If something must remain for recovery, it belongs in quarantine, not in the
  active execution path.

## Hard Stops

- Stop if cleanup cannot define one live execution path.
- Stop if the cleaned RM10 lane no longer runs at all.
- Stop if no minimal observable summary can be produced.
- Stop if the callable accelerator lanes remain too opaque to test honestly.
- Stop if thermal behavior completely swamps interpretation.
- Stop if every apparent effect collapses under simple falsification.
- Stop if the phase is drifting back into validator theatre or capability
  mythology.

## Required Outputs

- a short cleanup report
- a plain capability result
- a property map or explicit no-signal note
- a falsifier verdict
- a one-line recommendation among:
  - continue property science
  - continue handoff work
  - escalate lane work
  - terminate this line as empty
