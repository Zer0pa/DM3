# Phase 01.2.3.4.1.1.1 Context

## Locked Decisions

- This phase reopens heterogeneous compute as a bounded experimental
  hypothesis on the RM10-primary branch.
- The working hypothesis is not "heterogeneous compute proves DM3." The working
  hypothesis is narrower: a stable or interesting geometric-computation regime
  may require coupled CPU/GPU/NPU dynamics rather than emerging on a single
  lane first.
- Treat the RM10 as a candidate resonance chamber and organism environment,
  not merely as a validator target or replay endpoint.
- Treat the training document as a normative hypothesis source, not as proof.
- CPU, GPU, and NPU roles are to be tested as callable lane roles, not
  inherited from prose.
- Determinism, validator-default behavior, and canonical hashes are secondary
  telemetry in this phase unless they directly explain an observed effect or
  collapse.
- Prior `ABSTAIN` packets remain part of the inherited truth floor, but they do
  not veto a new bounded heterogeneity-first battery. They define the failure
  modes that this phase must attack directly.
- A negative result is acceptable. A fake win is not.
- Do not reduce this phase back into validator theatre.
- Do not mythologize "resonance" or "geometric computation." Every such claim
  must cash out into an observable, a battery, a boundary, or a falsifier.

## Mission Order

1. Translate the training document into testable hypotheses rather than
   inherited doctrine.
2. Re-map the current RM10 CPU, GPU, and NPU surfaces as actually callable or
   non-callable on the live branch.
3. Define a minimal heterogeneity-first observable contract that can be tested
   without requiring full canonical determinism first.
4. Run the smallest honest coupled-lane battery that contrasts CPU-only control
   with a heterogeneous candidate path.
5. Probe whether boundary, cache, packet, window, or environment choices change
   the response class.
6. Attack the result adversarially before any recommendation.

## Truth Floor Inherited

- Governed `F1` Genesis CPU control remains the only trustworthy governed phone
  lane retained so far.
- The top-level same-family `F2` surface is callable but unstable under the
  retained CPU/GPU/GPU/CPU bracket.
- No preserved same-family observable has survived honestly yet under the prior
  same-family-first doctrine.
- GPU has been exercised in bounded feasibility and residue-family probes, but
  not promoted into an accepted heterogeneous handoff.
- A receiptable DM3-facing NPU path still has not been proven.
- The capability-discovery reset on the cleaned single-lane surface closed as a
  no-signal / collapse packet on that contract.

These inherited truths remain visible. They do not answer the new question of
whether heterogeneity itself is part of the required operating regime.

## Primary Question

If the RM10 SoC is treated as a coupled resonance chamber with bounded
CPU/GPU/NPU roles, does any nontrivial geometric-computation behavior appear
that is absent or weaker on the CPU-only control?

## Subsidiary Questions

1. Does coupled lane choreography change behavior class rather than only speed?
2. Do boundaries, packet windows, cache locality, or handoff points carry the
   interesting behavior?
3. Does a heterogeneous path create or preserve response structure that the
   single-lane control does not?
4. Does the environment matter: persistent state, iterative streaming waves,
   resumed windows, or structured perturbation?
5. If NPU is not honestly callable, does a two-lane CPU/GPU chamber still show
   the relevant effect?
6. Does the apparent effect survive simple falsifiers, or does it collapse into
   logging, stale paths, timing artifacts, or thermal theatre?

## Scientific Stance

- "Resonance chamber" in this phase means a coupled dynamical execution surface
  whose lanes, caches, and handoffs may shape the evolution of a measurable
  state.
- "Geometric computation" in this phase means packet, token, or window updates
  that can be framed against lattice, boundary, graph, or field-like
  observables rather than only final file hashes.
- The training document's CPU scheduler, GPU relaxation, and NPU projection
  language is a role hypothesis to test.
- The phase is allowed to discover that the concept is empty, weak, narrow, or
  only partially realizable on the live RM10 surface.

## Working Hypotheses

- `H1`: a coupled heterogeneous path can produce a distinct response class from
  CPU-only control.
- `H2`: the interesting behavior, if any, lives disproportionately at
  boundaries, windows, and lane handoffs.
- `H3`: GPU participation may alter convergence, stability, or attractor class,
  not merely runtime.
- `H4`: even if the NPU is unavailable, a CPU/GPU chamber may still expose the
  relevant behavior.
- `H5`: the entire framing may collapse under bounded falsification.

## Minimal Observable Contract

Each serious battery must name:

- one anchor observable
- one drift observable
- one abstain rule

Admissible anchors for this phase include:

- response class: null, graded, thresholded, oscillatory, hysteretic,
  attractor-like, or collapse
- energy-like descent or an explicit `delta_E`-family proxy if surfaced
- coherence, ECC, spectral drift, or packet summary fields if the surface
  exposes them honestly
- pre/post handoff summary deltas
- lane-differential behavior under the same input family

Inadmissible anchors include:

- runtime alone
- heat alone
- GPU utilization alone
- validator or hash outcome by itself
- prose similarity or visual plausibility

## Battery Sequence

- Battery A: heterogeneous surface reality check and callable-lane map
- Battery B: heterogeneity-first micro-battery against CPU-only control
- Battery C: boundary and packet-window perturbation battery
- Battery D: organism-environment battery using persistent or resumed windows
- Battery E: NPU inclusion battery only if a callable path is found
- Battery F: adversarial falsification

Battery E is optional and must not be faked from hardware presence.

## Logging Discipline

Every serious run must preserve:

- run ID
- battery ID
- lane or split
- exact command
- cwd
- output path
- before/after battery state
- before/after thermal state
- observable summary
- boundary or handoff note when applicable

Determinism, validator, and hash outputs should be captured when available, but
remain secondary telemetry unless they directly explain the observed behavior.

## Hard Stops

- Stop if no callable heterogeneous candidate path exists at all.
- Stop if every observed effect collapses under simple controls or replay.
- Stop if the phase drifts back into treating hardware presence as evidence.
- Stop if the phase cannot name even one observable that distinguishes control
  from candidate behavior.
- Stop if thermal or timing noise dominates interpretation and cannot be
  separated.
