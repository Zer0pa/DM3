## Summary

Phase `01.1` should be executed as a split recovery phase, not as a resurrection
story and not as a source rebuild. The surviving evidence supports more than one
DM3 line across time.

The strongest source-backed split is:

1. a governed older Genesis / SoC lane with deterministic receipts and restart
   authority
2. a later hybrid DM3 line with real geometry, GPU, micro-transformer, and
   memory / scar residue, but weaker authority and missing source / asset
   continuity

The hybrid layer is real enough to study and partially runnable enough to test.
It is not honest to describe it as fully restored. The compiled Mac residue at
`/Users/Zer0pa/DM3/target/debug/dm3_runner` still runs a bounded contraction
micro-path and emits receipts, but several nominally different task names collapse
to the same `t1_contraction` surface with the same normalized receipt hash. That
means the current host executable is useful as executable residue and for
contamination testing, not as proof that boundary readout, holography, or R1R4
experiments are scientifically restored.

The best execution shape is a two-plan sequence:

1. inventory the artifact families and build a versioned architecture map
2. run bounded host micro-batteries, audit contamination, and issue an explicit
   authority verdict plus next-step recommendation
