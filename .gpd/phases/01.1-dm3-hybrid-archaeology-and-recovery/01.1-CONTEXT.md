# Phase 01.1 Context: DM3 Hybrid Archaeology and Recovery

## Mission

Perform a forensic archaeological recovery of the late DM3 hybrid layer on the
RedMagic 10 Pro and any linked recovery surfaces.

One governing question:

- what hybrid DM3 architecture existed
- what still survives
- what can be run now
- whether any surviving piece can be patched into a governed executable lane
  without lying about authority

This is not a literature review.
This is not a vibes exercise.
This is not a storytelling exercise.

## Required Sequence

1. find
2. classify
3. patch minimally
4. run what is honestly runnable
5. receipt everything
6. separate real recovery from memory, residue, and fantasy

If the surviving artifacts are sufficient for execution, run them.
If they are not sufficient for honest execution, return a precise engineering
plan instead of improvising a rebuild.

## Core Questions To Answer

### A. What was the recovered hybrid architecture?

Establish from surviving artifacts only:

- whether DM3 was implemented as a 3D double-Meru structure
- whether it was modeled as two 3D Meru / Sri-Yantra-like volumes joined at the
  bindu
- whether there was a center model, boundary-only transformer, nodewise
  micro-transformer logic, scar / memory / caching layer, geometry-only
  relaxation layer, or more than one hybrid design across time

Do not infer a single architecture if the evidence supports multiple
generations.

### B. How were learned components integrated?

Recover if present:

- center-model evidence
- HRM evidence
- model-size evidence
- boundary adapter evidence
- nodewise or micro-transformer evidence
- coupling / routing / bridge logic between geometry and learned modules
- whether learned modules were ornamental, assistive, or dominant

### C. What GPU path existed?

Recover if present:

- Vulkan kernels
- WGSL or shader assets
- GPU dispatch logic
- fused kernel traces
- runtime bindings
- compiled binaries
- loader scripts
- build scripts
- dependency assumptions
- CPU fallbacks
- cache or kernel-memory behavior discoverable from code or strings

Assume Vulkan before OpenCL unless the surviving artifacts prove otherwise.

### D. What is runnable now?

Determine:

- what binaries still exist on the RM10
- what scripts still exist
- what data, configs, and receipts still exist
- what executes directly
- what executes after minimal reversible patching
- what cannot execute without a longer rebuild

### E. What drift is fake, and what drift is real?

Investigate whether previous hash instability was caused by contamination from:

- date and time
- timestamps
- environment capture
- hardware metadata
- nondeterministic serialization
- host-specific paths
- locale
- timezone
- build provenance leakage

Do not hand-wave all drift away as timestamp noise.
Name the contamination surfaces exactly if they exist.

## Authority Rules

1. Source-backed evidence outranks narrative.
2. A runnable binary is not automatically governed evidence.
3. GPU existence is not parity.
4. Hybrid recovery is allowed only one interface at a time.
5. Every patch must be local, documented, reversible, and justified.
6. Running is required when honest.
7. If the job becomes a long engineering rebuild, stop and return a plan.

## Execution Behavior

### Phase 1: Inventory

Produce a hard artifact inventory.
For each artifact, capture:

- path
- type
- last modified time if available
- likely role
- confidence
- whether it is executable, inspectable, or only narrative residue

### Phase 2: Architectural Reconstruction

Build a versioned architecture map without forcing one story.
Use labels like:

- Hybrid A
- Hybrid B
- GPU experiment line
- center-model line
- nodewise line
- geometry-only line

For each line, record:

- evidence for it
- evidence against it
- what survived
- what is missing
- whether it was ever actually run

### Phase 3: Patch And Run

If an executable surface exists:

1. patch minimally
2. preserve originals
3. run micro-batteries first
4. capture receipts
5. compare outputs
6. write pass / fail / abstain honestly

Minimum acceptable run classes:

- launch / loader proof
- single micro run
- one controlled replay
- one CPU / GPU comparison if both are reachable

### Phase 4: Determinism / Contamination Audit

Return:

- definitely contaminating fields
- likely contaminating fields
- clean fields
- unresolved fields

### Phase 5: Verdict

State clearly which is true:

1. Recovered and runnable now
2. Recovered partially, runnable with bounded repair
3. Recovered only as compiled residue
4. Architecturally legible but not honestly runnable
5. Too fragmentary; requires a proper rebuild phase

## Hard Stops

Stop and return a blocker report if:

- the only GPU path depends on opaque unrecovered vendor behavior
- the surviving executable cannot be linked to a trustworthy receipt surface
- patching becomes a full rewrite
- the supposed hybrid layer cannot be separated from folklore
- something runs but the scientific claim it supports is unknowable
- the work requires a long engineering phase rather than bounded recovery

## GPU First-Move Rule

If a plausible Vulkan or GPU path is recovered on the RM10, the first move must
be a micro-battery only:

- prove the executable or runtime path is real
- prove kernel / pipeline / dispatch is callable
- prove outputs are capturable
- prove receipts can be preserved
- state whether parity comparison is possible yet

Not allowed as a first GPU move:

- broad rebuilds
- heroic optimization
- long benchmarks
- vague "it seems to work"
- expanding the scientific story because pixels moved on screen
