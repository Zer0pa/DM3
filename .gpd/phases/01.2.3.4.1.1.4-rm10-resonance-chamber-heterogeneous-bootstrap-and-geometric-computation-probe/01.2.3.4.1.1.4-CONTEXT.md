# Phase 01.2.3.4.1.1.4 Context

## Locked Decisions

- This is a heterogeneity-first probe on the RM10-primary branch.
- Do not wait for a fully stable core before testing whether heterogeneity is
  itself part of the condition for structured behavior.
- RM10 is the primary instrument. Mac remains a control and support lane.
- Treat the SoC as a possible resonance chamber only as an experimental
  hypothesis, not as an earned fact.
- Keep validator-default behavior, bitwise determinism, and canonical hash
  agreement as telemetry and governance context, not as the front-door stop for
  this phase.
- A GPU-backed or opaque accelerator path is not automatically an explicit
  heterogeneous result. The phase must still name lane ownership, handoff
  points, and observables honestly.
- The word "organism" is permitted only as an engineering shorthand for a
  stateful environment with controlled inputs, memory surfaces, and feedback
  loops. It does not license metaphysical claims.
- If no callable user-space NPU path exists, the phase must say so explicitly
  and continue with CPU/GPU-first experiments rather than inventing NPU work.
- Do not let old validator doctrine or same-family abstain doctrine prohibit
  exploratory heterogeneity-first batteries. Do let them cap the claim ceiling.

## Mission Order

1. Freeze the resonance-chamber question as a bounded scientific and
   engineering hypothesis.
2. Define the smallest observable contract that can compare CPU-only,
   CPU/GPU-coupled, and NPU-assisted attempts honestly.
3. Map the live RM10 environment: CPU lane, GPU lane, NPU or DSP-adjacent
   callable surfaces, memory and thermal state, and the active runner paths.
4. Reuse existing capture tooling where possible and write only the minimum new
   harness needed for a heterogeneity-first battery.
5. Run one CPU control and one CPU/GPU resonance-oriented bracket on the
   attached RM10 with stronger timeout and periodic telemetry.
6. Attempt one bounded NPU projection or priming path only if a callable
   user-space surface is real.
7. Falsify the result aggressively against timeout, thermal, logging, and
   residue-family illusion.

## Primary Question

Can heterogeneity itself create, preserve, or clarify a structured observable
family on RM10 that CPU-only work misses or destroys?

## Subsidiary Questions

1. What does it operationally mean to treat the RM10 SoC as a resonance
   chamber rather than a conventional deterministic pipeline?
2. Which currently exposed observables can stand in for the training
   document's stronger claims about energy descent, coherence, ECC, and
   spectral stability?
3. Does a CPU/GPU-coupled bracket exhibit a stable response class, threshold,
   cluster, or persistence pattern that differs from CPU-only control?
4. Is the closing `cpu_b` failure a timeout artifact, a late-session thermal or
   process issue, or a genuine dynamical instability?
5. Is there any callable NPU projection path at all, and if so does it alter
   the observable family or only latency?

## Working Model

The phase treats the RM10 as a coupled dynamical environment with three
possible roles:

- CPU: scheduler, boundary owner, runloop arbiter, telemetry and acceptance
- GPU: high-throughput nonlinear update or relaxation surface
- NPU: optional projection, priming, or proposal generator if a callable path
  exists

The phrase "resonance chamber" means the meaningful output may live in the
trajectory class of the coupled system under repeated forcing, not only in one
final deterministic output file.

This does not assert literal physical resonance inside the phone. It asserts a
testable computational hypothesis: lane coupling may alter attractor behavior,
completion class, or stability class in a way that matters scientifically.

## Minimal Observable Contract

Every serious run in this phase must name:

- one anchor observable
- one drift observable
- one abstain rule

The default anchor set for this phase is:

- `delta_E` when exposed
- `coherence` when exposed
- row completion class: `completed`, `timeout`, `zero_byte_receipt`, or
  `other_failure`
- receipt-bearing output size and path

The default drift set is:

- low/high cluster flips across repeated rows
- closing-row timeout or receipt dropout
- temperature or thermal-status excursions correlated with the class flip
- lane-marker mismatch such as missing GPU initialization or missing CPU force
  marker

Default abstain:

- lanes are not actually comparable
- the output family is opaque
- NPU remains inventory-only
- telemetry is too thin to localize drift

## Signal Standard

Signal in this phase means one or more of:

- reproducible CPU versus CPU/GPU class separation
- reproducible low/high cluster structure
- timeout disappearance under a controlled timeout increase
- persistent timeout under stronger capture, implying a real late-session
  defect
- lane-dependent change in `delta_E`, `coherence`, or completion class
- a callable NPU assist path that produces a receiptable, comparable boundary

GPU heat, runtime length, library presence, or device marketing do not count as
signal by themselves.

## Hard Stops

- Stop if no callable GPU-bearing surface exists on the attached device.
- Stop if the capture harness cannot retain exact commands, cwd, thermal state,
  and row outcomes.
- Stop if the GPU-bearing surface is only a legacy substitute rather than the
  top-level family under test.
- Stop if the NPU story is still only hardware inventory after a direct path
  search.
- Stop if the only apparent effect is explained entirely by timeout wrappers,
  output-path corruption, or thermal collapse.
