## Mathematical Framework

Treat the phase as a coupled discrete-time dynamical system over a state
surface `x_t` with lane-shaped update operators:

- CPU scheduler and boundary owner: `S`
- GPU relaxation or nonlinear update operator: `G`
- optional NPU proposal or projection operator: `P`

The experiment does not need the training document's full ontology to be true.
It only needs the following testable model:

`x_(t+1) = S(G(P(x_t, u_t), u_t), telemetry_t)`

where:

- `u_t` is the forcing or input window
- the output of interest is the trajectory class under repeated forcing
- the meaningful comparison is not only final file equality, but the observable
  family over the trajectory

For current branch surfaces, the strongest measurable stand-ins are:

- `delta_E`
- `coherence`
- completion class and timeout behavior
- retained output bytes and receipt paths
- lane markers from stdout
- thermal state and process survival during the run

This is much weaker than the training document's full claims about ECC,
Laplacian tension, isotropy, and synchronization, but it is honest for the
current device surfaces.

## Standard Approaches

### 1. Physical reservoir computing as design pattern

Use reservoir-computing literature as an experiment-design template:

- high-dimensional nonlinear substrate
- repeated forcing
- state trajectory as the computational object
- measure memory, synchronization, stability, and separability rather than only
  exact end-state equality

For this phase, that means comparing CPU-only and CPU/GPU-coupled run classes
under the same input family and row order.

### 2. Coupled-oscillator and synchronization language as operational metaphor

The training document invokes mode-locking, synchronization, and resonance.
Operationally, the safe extraction is:

- look for thresholded stability or instability
- look for repeated cluster structure
- look for persistence versus collapse across repeated rows
- look for lane coupling that changes the completion or cluster class

Do not claim literal Kuramoto dynamics unless the measured observables justify
that level of interpretation.

### 3. Geometric and diffusion-style computation

Message passing, diffusion, and manifold computation give a more grounded
picture than mystical resonance language:

- the lattice or graph is the geometry
- each lane implements some part of a repeated update rule
- a useful heterogeneity-first question is whether GPU or NPU participation
  changes the effective update class rather than only the speed

### 4. Environment-first execution

The training document implies that the system is stateful and environment
dependent. For this phase, "environment" means:

- working directory and sidecar availability
- input packet shape
- output directory discipline
- battery and thermal state
- cacheable or repeated row order
- process survivability across a multi-row bracket

The safe scientific interpretation is:

- the SoC may behave as a resonance chamber if lane coupling and repeated
  forcing create stable or unstable trajectory classes that are not reducible
  to a single-lane control narrative
- the chamber is computational, not mystical

This interpretation becomes stronger only if the phase can show:

- repeated lane-sensitive clusters
- threshold or persistence behavior
- timeout or instability localization that tracks the coupled bracket

## Validation Strategies

### Minimal honest battery

1. Environment manifest
   - retain current CPU, GPU, and NPU-adjacent callable surfaces
   - retain battery, thermal, storage, process, and working-directory state

2. CPU control anchor
   - run one hardened `F1` CPU control packet
   - keep exact command, cwd, receipts, and thermal snapshots

3. CPU/GPU resonance bracket
   - rerun the top-level `F2` CPU/GPU/GPU/CPU bracket with longer timeout and
     periodic telemetry
   - measure `delta_E`, `coherence`, completion class, receipt presence, and
     thermal drift

4. NPU projection probe
   - search for a real user-space callable path
   - if callable, run one bounded proposal or assist micro-probe
   - if not callable, return explicit `ABSTAIN`

5. Falsifier
   - attack timeout artifact, thermal confound, stale-path substitution, and
     output corruption

### Acceptance shape

The phase does not need a pass narrative. It needs one decisive truth among:

- CPU/GPU coupling changes the observable family in a repeatable way
- CPU/GPU coupling does not change the observable family meaningfully
- the apparent effect is an artifact of timeout, thermal drift, or path
  substitution
- NPU is still non-callable and remains `ABSTAIN`
