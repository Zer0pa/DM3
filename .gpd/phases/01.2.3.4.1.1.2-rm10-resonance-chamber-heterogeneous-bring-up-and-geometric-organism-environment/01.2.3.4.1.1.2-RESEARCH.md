## Mathematical Framework

The most useful formal interpretation for this phase is not "prove the training
doc."
It is to cast the proposed architecture as a bounded coupled dynamical system.

Working formalism:

1. state lives in a retained packet or field-like representation emitted by the
   current binaries
2. input or perturbation changes the packet or lane configuration
3. a fast update operator acts on the state
4. measurements are taken on exposed observables after each window
5. meaningful behavior, if any, appears as structured transients, thresholds,
   cluster changes, persistence, or synchronization-like regimes

That puts the phase closest to:

- reservoir-computing logic for transient state reuse
- coupled-oscillator logic for synchronization and regime change
- geometric / graph-based computation for non-Euclidean state organization

The training document's language about contraction, descent, coherence, ECC,
and synchronization should therefore be treated as candidate observables over a
coupled update system, not as already-proven physics.

## Standard Approaches

### 1. Heterogeneity-first coupled-lane mapping

Map current branch surfaces into roles:

- `F1` `genesis_cli` CPU lane -> scheduler, gating, control comparator
- top-level `F2` `/data/local/tmp/dm3_runner` -> coupled chamber candidate with
  known CPU/GPU/GPU/CPU bracket
- optional NPU / DSP path -> projection or priming only if a real callable
  surface is found
- memory and packet boundaries -> persistence medium

This is the best match to the training document while remaining honest about
what the branch can run.

### 2. Organism-environment construction

Instead of treating each command invocation as the whole experiment, define one
bounded environment:

- fixed asset set
- fixed `cwd`
- explicit perturbation windows
- explicit run order
- explicit packet and telemetry capture

The "organism" is then the evolving packet family or session-level state under
those controlled perturbations.

### 3. CPU control plus coupled comparison

Use CPU-only control as a comparator, but not as the admission gate for the
phase.
The decisive comparison is whether coupled-lane execution changes response
class in a structured way.

### 4. Optional assist-stage discovery

If a true NPU path is not presently callable, do not block the phase.
Treat `CPU+GPU` as the first real chamber and keep NPU as an auxiliary probe.

## Common Pitfalls

### 1. Opaque-lane inflation

The most likely mistake is to narrate hidden in-binary GPU or NPU work as an
explicit stage-owned handoff.

Mitigation:

- require pre-handoff and post-handoff artifacts if stage ownership is claimed
- otherwise classify the result only as coupled behavior, not explicit role
  partition

### 2. Thermal theatre

A resonance-chamber framing can easily drift into "the phone got hot, therefore
something interesting happened."

Mitigation:

- thermal traces are secondary telemetry only
- no signal claim without paired observable change

### 3. Legacy substitution

The legacy `/data/local/tmp/dm3/dm3_runner` surface is callable and tempting.

Mitigation:

- use it only as a control or contrast surface
- do not silently substitute it for the top-level family

### 4. NPU mythology

Vendor libraries and daemons are not a usable lane.

Mitigation:

- a real NPU claim requires a callable command, bounded role, and retained I/O

### 5. Validator regression drift

This phase is not primarily about validator-default repair, but validator
behavior can still contaminate interpretation if it mutates packet handling or
config flow.

Mitigation:

- keep validator-default as telemetry or boundary condition only
- do not let it dominate the experiment design

## Validation Strategies

### Primary validation

- compare CPU-only versus coupled-lane response on one named observable family
- repeat both control and candidate rows
- preserve exact command, `cwd`, packet paths, telemetry, and asset identity

### Adversarial validation

- stale-path and `cwd` dependence
- config mutation or asset mutation across rows
- timeout artifacts
- zero-byte or truncated output paths
- thermal covariance without observable change

### Negative result standard

The phase should terminate cleanly as negative if:

- coupled-lane behavior stays semantically equivalent to CPU control
- any apparent difference collapses under repeat or falsifier checks
- the coupled chamber cannot be defined without legacy substitution or opaque
  stage ownership
