# Phase 01.2.3.4.1.1.3.1 Context

## Locked Decisions

- This is the live continuation phase after `01.2.3.4.1.1.3`.
- The scope is same-family width-boundary environment engineering on the
  top-level `/data/local/tmp/dm3_runner` surface only.
- Do not reopen chamber-science, heterogeneous role-partition, or NPU
  execution claims in this phase.
- Keep `F1` and validator-default work outside the primary battery unless
  needed as support telemetry.
- The governing question is whether the retained `cpu_b` break is a
  `180`-second wrapper ceiling, a late-session hang, or receipt/output-path
  corruption under the same-family replay.
- Any positive story must come from preserved same-family tuples, not from
  runtime length, GPU initialization, or device heat.

## Mission Order

1. Freeze the narrow observable contract for same-family width-boundary work.
2. Extend the existing top-level `F2` capture harness only as needed for
   longer timeout, periodic telemetry, and bounded row-set replay.
3. Run one exact four-row same-family replay with stronger timeout and
   periodic telemetry on the attached RM10.
4. Run one bounded follow-up replay only if needed to classify whether the
   final `cpu_b` defect is session-width-sensitive or row-local.
5. Falsify the result against wrapper-ceiling, thermal, stale-path, and
   receipt-corruption explanations.
6. Return one bounded verdict and one exact next move.

- The top-level same-family surface on `/data/local/tmp/dm3_runner` is real and
  callable.
- A bounded two-window smoke on that same family survived honestly.
- The widened four-window replay collapsed before interpretable tuples
  survived.
- The strongest retained result is `boundary_sensitivity_without_persistence`.
- NPU remains `ABSTAIN / inventory_only`.
- The live justified next move is environment engineering on the same-family
  width boundary.

## Primary Question

Can the same-family four-row replay survive under a stronger environment
envelope long enough to decide whether the final `cpu_b` break is a wrapper
ceiling artifact or a real late-session defect?

## Subsidiary Questions

1. Does the exact four-row replay complete when row timeout is increased and
   periodic telemetry is retained?
2. If the full replay still fails at `cpu_b`, does an isolated `cpu_b` replay
   complete on a fresh session?
3. Do retained receipts and telemetry support a width-boundary interpretation,
   a row-local defect interpretation, or only an unresolved collapse?
4. Is the failure better explained by thermal drift, stale-path substitution,
   or output corruption than by the session structure itself?

## Minimal Observable Contract

Every serious run in this phase must name:

- one anchor tuple
- one drift tuple
- one abstain rule

The default anchor tuple is:

- `delta_E`
- `coherence`
- row completion class: `completed`, `timeout`, `zero_byte_receipt`, or
  `other_failure`
- retained receipt path and size

The default drift tuple is:

- change in `cpu_b` completion class under a stronger timeout
- change in receipt survival under exact same-family replay
- row-local versus session-local failure split
- thermal or process-state excursions correlated with collapse

Default abstain:

- tuple family is still missing or zero-byte on decisive rows
- the follow-up replay is not comparable to the primary replay
- the capture path silently drifts off the top-level family

## Hard Stops

- Stop if the attached device is not reachable over ADB.
- Stop if the tool changes would introduce a new launcher surface or a hidden
  binary substitution.
- Stop if the replay cannot retain exact commands, cwd, row order, and receipt
  custody.
- Stop if the only surviving evidence is runtime noise without tuples.
