## Existing Evidence To Leverage

- The top-level `dm3_runner` family can emit a bounded same-family two-window
  packet with comparable CPU and GPU tuples.
- The widened four-row packet collapsed before those tuples survived.
- The older official rerun already suggested the decisive break is the closing
  `cpu_b` row under a `180`-second ceiling.
- Thermal status stayed flat in the retained widened packet, so simple thermal
  collapse is not the leading explanation.

## Engineering Hypotheses

### H1. Wrapper-Ceiling Artifact

The four-row packet is fundamentally viable, and the decisive `cpu_b` break
was caused mainly by the `180`-second row timeout.

### H2. Late-Session Hang

The closing `cpu_b` row enters a real late-session stall that persists even
when the wrapper ceiling is relaxed.

### H3. Receipt Or Output Corruption

The decisive break is not compute completion itself but receipt generation or
pull-path corruption near session end.

### H4. Width-Boundary Sensitivity

The defect is session-local: a fresh isolated `cpu_b` replay can survive while
the exact four-row packet still fails.

## Minimal Experimental Design

### 1. Exact Replay Under Stronger Envelope

Run the same top-level four-row CPU/GPU/GPU/CPU replay with:

- longer row timeout
- periodic telemetry samples
- preserved exact row order
- preserved same binary, cwd, and output discipline

This decides whether the old ceiling was merely too short.

### 2. Bounded Follow-Up Replay

If the exact replay still fails decisively at `cpu_b`, run one isolated
follow-up replay for `cpu_b` only on a fresh session using the same binary and
capture discipline.

This separates row-local failure from session-width sensitivity without opening
new families.

### 3. Artifact-Based Falsification

Attack the result against:

- wrapper-timeout artifact
- stale-path substitution
- thermal drift
- receipt corruption
