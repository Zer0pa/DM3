# Phase 01.2.3.1 Research

## Objective

Research how to execute the branch phase
`01.2.3.1 RM10 Primary-Platform Engineering, History Mining, And Heterogeneous Battery Bootstrap`
under the active hypothesis branch
`hypothesis/rm10-primary-platform-heterogeneous-learning`.

This phase is not trying to prove a noun for DM3. It is trying to build a
clean RM10-first engineering and scientific learning program that can:

1. make the Red Magic 10 Pro a governed primary instrument,
2. mine the older device-side histories without inheriting them as authority,
3. stand up explicit CPU, GPU, NPU-feasibility, and heterogeneous tracks, and
4. run early branch batteries that can strengthen or kill the branch
   hypothesis with receipts.

## Branch Constraint

This phase is being planned on a hypothesis branch. The main line after
`01.2.3` points to launcher-adjacency provenance or explicit redevelopment
classification. This branch tests a different question:

- whether RM10-primary engineering yields a more informative evidence surface
  than launcher-first continuation, and
- whether heterogeneous role partition is scientifically useful early rather
  than only after single-lane closure.

The branch still inherits the anti-proxy rules from the restart:

- no proof narrative,
- no hardware-presence inflation,
- no undeclared redevelopment,
- no treating pretty plots or warm silicon as evidence.

## Historical Mining Frame

The older histories matter because they contain executable clues, observables,
failure modes, and role-partition hints that the present branch can test.
They do not matter as authority.

Two history families must be mined separately:

1. **Governed SoC / Genesis surface**
   - strongest signal for run discipline, receipt expectations, and repeatable
     command identity
   - likely source of the most honest CPU control lane on device

2. **Later DM3 prototype residue**
   - contains real device-side experimentation, bundled assets, GPU attempts,
     boundary-oriented tasks, and structured experiment traces
   - weaker authority, but high value for clues: names, partitions,
     observables, failure signatures, and asset expectations

Mining deliverables should therefore be indexes, not inherited semantics:

- clue ledger
- observable index
- role-partition index
- failure-pattern index

## Primary-Source Scientific Anchors

These sources are prompts for what to test. They are not proofs that DM3 is the
same kind of object.

### Heterogeneous mobile compute

1. **Characterizing Mobile SoC for Accelerating Heterogeneous LLM Inference**
   (`arXiv:2505.07023`)
   - abstract states that mobile SoCs with shared memory across CPU, GPU, and
     NPU can benefit from heterogeneous inference, but only if synchronization
     and partition costs are handled explicitly
   - branch implication:
     heterogeneous execution should be treated as a role-partition problem, not
     as an article of faith about accelerators

2. **PointSplit: Towards On-device 3D Object Detection with Heterogeneous
   Low-power Accelerators** (`arXiv:2504.03654`)
   - shows that mixed mobile GPU and EdgeTPU execution can be viable when work
     is partitioned by role
   - branch implication:
     the RM10 battery should include stage-wise and representation-wise splits,
     not just "CPU vs GPU"

### Boundary-special computation

3. **Hybrid Boundary Physics-Informed Neural Networks**
   (NeurIPS 2025, OpenReview)
   - explicitly distinguishes interior and boundary treatment
   - branch implication:
     boundary-only, interior-only, seam, and partial-boundary masking batteries
     are scientifically justified early

### Spectral / modal behavior

4. **Neuro-Spectral Methods for Solving Initial Value Problems in ODEs**
   (`arXiv:2506.22213`)
   - highlights spectral bias and causality/ordering issues in standard PINN
     approaches
   - branch implication:
     the RM10 branch should not trust low-frequency or averaged summaries alone;
     it should probe mode sensitivity, settle patterns, and causal ordering

### Operator / observable reduction

5. **A Hybrid Framework for Efficient Koopman Operator Learning**
   (`arXiv:2504.18676`)
   - motivates searching for reduced observable spaces and linearizable
     propagators
   - branch implication:
     one battery should look for small observable sets or transformed summaries
     that preserve predictive structure better than raw state dumps

### Topology-sensitive regime detection

6. **Quantum Barcodes: Persistent Homology for Quantum Phase Transitions**
   (`arXiv:2504.10468`)
   - shows persistent-homology changes can track regime changes

7. **Topological Detection of Hopf Bifurcations via Persistent Homology**
   (`arXiv:2603.27395`)
   - shows time-series topology can detect dynamical transitions without
     explicit governing equations

8. **A Stable Measure of Chaos in Dynamical Systems Using Persistent Homology**
   (`arXiv:2601.10900`)
   - motivates topology-aware measures that are more stable to perturbation
     than naive local distances

Branch implication:

- topology is a real candidate summary layer for branch batteries, especially
  around regime change, failure onset, or hardware-role changes

### Reciprocity

9. **Enforcing Reciprocity in Operator Learning for Seismic Wave Propagation**
   (`arXiv:2602.11631`)
   - hard-codes swap symmetry between source and receiver and reports both
     physical-consistency and performance benefits
   - branch implication:
     explicit source/receiver, seam-side, or mirrored-role swap batteries are
     scientifically justified if analogous paired roles exist in DM3

### Persistence / memory discipline

10. **A practical guide to reservoir computing with physical dynamical systems**
    (`arXiv:2403.20242`)
    - sharpens the distinction between a rich physical substrate and a
      substrate that can actually be controlled, benchmarked, and read out
    - branch implication:
      persistence batteries must test usefulness, controllability, and residue
      decay, not just whether the system "has memory"

## Planning Implications

### What the phase must do before serious execution

The internal pack is not optional. Before any branch run is promoted as serious
evidence, the phase must write:

- one internal engineering PRD
- one heterogeneous compute plan
- one old-history mining plan
- RM10 CPU/GPU/NPU/heterogeneous runbooks
- boundary / spectral / topology / reciprocity / persistence /
  hardware-role battery docs
- Comet schema and ledger rules
- thermal and checkpoint policy
- kill criteria and stopping rules
- history mining indexes

This is not document theatre. It is what prevents branch execution from turning
into opportunistic probing with no interpretable contract.

### Recommended phase decomposition

The phase naturally splits into four execution plans:

1. **Internal pack and branch governance**
   - write the engineering PRD, hetero plan, history mining plan,
     Comet/ledger rules, thermal/checkpoint policy, and kill criteria

2. **History mining and role / observable extraction**
   - mine governed SoC and later DM3 residues into clue, observable, role, and
     failure indexes

3. **RM10 system engineering and runbook build-out**
   - map `/data/local/tmp`, wrappers, binaries, assets, telemetry, shell
     assumptions, and CPU/GPU/NPU / heterogeneous run classes

4. **Battery matrices plus first receipted RM10 passes**
   - write battery docs
   - run bounded CPU control
   - run GPU feasibility and NPU feasibility probes
   - run one explicit heterogeneous comparison if and only if the control lane
     and instrumentation are stable enough
   - merge outputs into a 10-document science/engineering pack

### Recommended wave structure

- **Wave 1:** plans 1 and 2 in parallel
  because internal governance docs and history mining can proceed independently
- **Wave 2:** plan 3
  because runbooks and execution surfaces depend on Wave 1 outputs
- **Wave 3:** plan 4
  because battery execution should consume the governance docs and runbooks

## First Observable Recommendation

Do not begin with the retired bundled `G2` family.

For the first common observable family, prefer a bounded RM10 control that is:

- receiptable from the current live surface,
- small enough for repeated CPU runs,
- comparable across staged variants,
- meaningful under thermal and checkpoint logging, and
- not dependent on an unrecovered launcher story.

Candidate family:

- a CPU-controlled geometry / settle / readout micro-battery anchored to the
  governed SoC / Genesis lane or another already-live device-side control
  executable
- then explicit GPU and NPU feasibility probes that either accelerate,
  decorate, or fail against that same control family

If no such common family can be stabilized, the branch should record that as a
real failure rather than widening into substitute tasks.

## Risks and Kill Conditions

The planner should carry these risks into explicit forbidden proxies and
acceptance tests:

- GPU visibility gets mistaken for GPU scientific readiness
- NPU / DSP sensor visibility gets mistaken for a usable compute path
- history mining silently becomes history inheritance
- the branch measures only performance, not interpretability or run identity
- heterogeneity is declared useful without a common control observable
- phone heat or busyness is treated as evidence of scientific significance

Kill the branch path early if:

- the branch cannot define one common observable family for CPU / GPU / NPU /
  heterogeneous comparison,
- device staging or identity cannot be kept reproducible,
- heterogeneity only works by making outputs opaque,
- history mining adds labels but not actionable constraints,
- or the branch starts depending on undeclared redevelopment.

## Recommended Verification Focus

Planning and execution should explicitly verify:

- command identity and staging surface (`/data/local/tmp` versus shared storage)
- thermal / battery / memory pressure snapshots before and after runs
- receipt emission and checkpoint identity
- common-observable comparability across compute lanes
- abstain conditions when GPU or NPU are merely reachable but not interpretable
- explicit branch verdicts per hypothesis family:
  strengthen, collapse, or remain unjudged
