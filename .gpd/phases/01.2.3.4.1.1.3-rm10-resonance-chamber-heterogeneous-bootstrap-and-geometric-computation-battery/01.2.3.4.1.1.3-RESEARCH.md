# Phase 01.2.3.4.1.1.3: RM10 Resonance-Chamber Heterogeneous Bootstrap And Geometric Computation Battery - Research

**Researched:** 2026-04-06
**Domain:** exploratory computational physics / mobile heterogeneous systems / dynamical-computation experiment design
**Confidence:** MEDIUM

## Summary

The phase should be planned as a bounded dynamical-systems experiment on the
RM10, not as another validator or parity campaign. The training document's
CPU/NPU/GPU split is useful only if translated into an operational method:
CPU runs scheduling, perturbation control, measurement, and artifact custody;
GPU is treated as a callable relaxation or coupled-update surface where it is
already exposed by existing binaries; NPU is treated as a bounded projection /
priming assist only if a user-space path can be named and receipted. The phase
must compare chamber behavior against CPU-only control, but CPU-only stability
is not the admission gate for opening the chamber battery.

The right primary path is not "prove the theory" and not "force explicit
heterogeneous handoff." It is to build a minimal geometric organism
environment, run repeated perturb-and-relax windows on callable RM10 surfaces,
and ask whether heterogeneous coupling changes behavior in any honest
observable family. The cheapest honest observables on current surfaces are
packet summary tuples and already-emitted dynamical fields such as delta /
descent traces, coherence-like fields, mode clusters, persistence class,
threshold or hysteresis class, and boundary-perturbation response. Treat them
as empirical proxies, not as proof that the training document's intended
kernel already exists.

**Primary recommendation:** Plan around a CPU-only versus CPU/GPU-coupled
windowed chamber battery on existing RM10 surfaces, in parallel with a sharp
NPU assist-path probe. Use the training document as a role-partition
hypothesis, not as a success criterion.

## Standard Approaches

### Approach 1: Existing-Surface Chamber Battery (RECOMMENDED)

**What:** Build a minimal persistent environment on the live RM10 surfaces and
run a repeated-window CPU-only versus CPU/GPU-coupled comparison using existing
callable binaries and helper scripts.

**Why standard enough:** It respects the current branch evidence floor, avoids
inventing a hidden accelerator stack, and turns heterogeneity into a direct
experimental variable.

**Key steps:**

1. Freeze one live environment root and one output root for the phase.
2. Implement or adapt a helper that runs repeated windows with explicit
   perturbation schedules and retains comparable packet summaries.
3. Run A/A CPU-only windows first as a control, but do not require them to be
   fully "stable-core complete" before chamber testing.
4. Run CPU/GPU-coupled windows on the same or nearest honest observable family.
5. Measure delta traces, persistence class, hysteresis class, and boundary
   sensitivity.
6. Attack the apparent effect with null perturbation, stale-path, and thermal
   falsifiers.

**Known difficulties:**

- `dm3_runner` is callable but unstable, so chamber batteries must stay small.
- GPU-backed work may be opaque internal use rather than explicit split.
- observable fields may be weak and require classification rather than exact
  equality.

### Approach 2: Environment-First CPU/GPU Differential Probe (FALLBACK)

**What:** If same-family chamber comparison is too unstable, build the
environment and test whether controlled config / window mutations cause
different response classes on the CPU control lane and on the nearest callable
GPU-backed residue family, without claiming the families are identical.

**When to switch:** when the exact same-family battery remains blocked by the
`F2` instability or by missing comparable receipts.

**Tradeoffs:** lower claim strength, but still useful for deciding whether the
heterogeneity-first hypothesis is worth further engineering.

### Approach 3: Immediate NPU Assist Opening (SECONDARY, NOT PRIMARY)

**What:** Try to identify one real user-space NPU or DSP-assisted projection /
priming role now.

**When to use:** in parallel with Approach 1, but never as the only line of
work.

**Tradeoffs:** high upside if callable; high probability of `ABSTAIN` because
current branch evidence does not yet expose a real path.

### Anti-Patterns to Avoid

- **Waiting for a fully stable core before any chamber work:** this simply
  reinstates the assumption the phase is meant to test.
- **Treating opaque internal accelerator use as explicit heterogeneous role
  partition:** the branch has already been burned by this.
- **Calling runtime or heat a resonance effect:** those are confounds unless
  paired with behavioral change.
- **Trying to prove the training document:** the document should drive
  perturbations, roles, and observables, not define success.

## Common Pitfalls

- **False heterogeneity claim:** a binary that internally touches GPU is not the
  same as an explicit CPU->GPU handoff.
- **Thermal aliasing:** repeated windows can create fake persistence or drift if
  temperature ramps with each run.
- **Config mutation:** `genesis_cli` has already shown in-place config mutation.
- **Output contamination:** reused output roots can look like persistence.
- **Family substitution:** if same-family comparison breaks, do not silently
  switch to another family and keep the same claim.
- **Ontology inflation:** phrases like `coherence`, `ECC`, or `Laplacian`
  should stay as candidate observables unless a current surface actually emits
  them.

## Key Equations and Starting Points

Start from these experimental definitions, not from a claimed field theory:

- chamber state update:
  - `x_{k+1} = F_L(x_k, u_k, e_k)`
- observable tuple:
  - `o_k = (o_{k,1}, ..., o_{k,n})`
- per-window drift:
  - `Δo_k = o_k - o_{k-1}`
- lane contrast distance:
  - `D_L = d(o^{(L)}_k, o^{(CPU)}_k)`
- hysteresis score:
  - `H(λ) = d(o^{up}(λ), o^{down}(λ))`

The planner should keep the mathematics at this level unless a task is
explicitly about deriving a richer model from data.
