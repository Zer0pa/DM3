# Phase 01.2.3.4.1.1.3 Context

## Locked Decisions

- This phase is a heterogeneity-first phase on the RM10-primary hypothesis
  branch.
- The phase does not wait for a perfectly stable CPU-only core before testing
  coupled heterogeneous execution.
- CPU-only control remains necessary as a contrast lane and authority anchor,
  but it is not the front-door gate for opening heterogeneous experiments here.
- Treat the RM10 SoC as a candidate resonance chamber rather than only as a
  conventional serial pipeline.
- Interpret "resonance chamber" operationally: coupled CPU scheduling,
  GPU-parallel relaxation, optional NPU projection or priming, shared memory /
  cache pressure, persistent state, and repeated bounded perturbation.
- Build and test a minimal "geometric organism environment" rather than only
  isolated one-shot commands.
- Determinism, validator-default repair, and canonical hash alignment remain
  important telemetry, but they are not the phase-defining question.
- No mythologizing. Every claimed effect must reduce to a named observable,
  perturbation, lane choice, or falsifier result.
- No hardware-marketing inference. GPU or NPU presence alone does not count as
  a lane.
- The sovereign claim gate still stands: exploratory success here does not
  become a governed authority pass without preserved observables and receipts.

## Mission Order

1. Freeze the new doctrine into a PRD and research pack.
2. Treat the training document as a hypothesis source, not proof.
3. Turn the RM10 into a minimal coupled environment with persistent state,
   bounded ingress, and repeated perturbation windows.
4. Run the smallest honest resonance-oriented batteries on callable CPU/GPU
   paths first.
5. Open an NPU assist role immediately if and only if a real user-space path
   can be named; otherwise classify the missing path explicitly and continue
   with CPU/GPU-coupled work.
6. Measure whether heterogeneity changes behavior, not merely speed.
7. Attack any apparent signal with hostile falsifiers before narrating value.

## Reframed Primary Question

If geometric computation on the RM10 is chamber-like rather than single-lane,
can a bounded heterogeneous environment produce a measurable, falsifiable
response family before a stable CPU-only core exists?

## Secondary Questions

1. What does the training document's CPU / GPU / NPU role partition mean as a
   runnable experiment rather than a manifesto?
2. Which callable lane roles exist now on RM10: scheduling, projection,
   relaxation, verification, persistence?
3. Does coupling change response class, persistence, hysteresis, boundary
   sensitivity, or spectral behavior relative to CPU-only control?
4. Which observables are cheap enough and honest enough to track per window on
   today's surfaces?
5. Does the phase uncover a genuine chamber effect, a weaker accelerator-only
   speed effect, or no effect at all?

## Working Hypotheses

- `H1`: Some DM3-relevant signal appears only, or more sharply, when the RM10
  is treated as a coupled CPU/GPU chamber instead of a CPU-only lane.
- `H2`: The most informative early observables are dynamical rather than
  canonical: descent traces, coherence proxies, spectral or cluster stability,
  phase transitions, boundary retention, or persistence under repeated forcing.
- `H3`: A minimal geometric environment matters: retained state, repeated
  windows, bounded ingress, and explicit perturb-and-relax cycles may expose
  structure that one-shot invocations miss.
- `H4`: If a real NPU path exists, its honest early role is bounded projection,
  priming, or tagging, not opaque whole-pipeline authority.
- `H5`: The chamber story may still collapse under falsification, and that is a
  valid phase outcome.

## Chamber Interpretation

This phase treats the RM10 as a candidate coupled dynamical substrate with:

- CPU as scheduler, perturbation controller, verifier, and artifact custodian
- GPU as parallel relaxation or field-update surface where callable
- NPU as projection / priming assist only if callable from user space
- persistent shared state across windows
- bounded perturbation waves rather than one-shot pass/fail executions
- environment variables, caches, output roots, and thermal conditions treated
  as part of the experimental environment rather than incidental noise

## Minimal Observable Contract

Each serious battery must define:

- one anchor observable
- one drift observable
- one abstain rule

Admissible anchor candidates in this phase include:

- energy-like descent or delta trace
- coherence or boundary-window proxy already exposed by the current surfaces
- stable packet summary fields or comparable receipt tuples
- threshold, mode-flip, hysteresis, or persistence classification
- spectral or cluster classification over repeated windows

Inadmissible anchors include:

- runtime alone
- GPU utilization alone
- thermal rise alone
- visual plausibility
- prose similarity

## Immediate Battery Direction

- Battery A: callable chamber inventory and reusable surface map
- Battery B: minimal geometric organism environment bring-up
- Battery C: CPU-only versus CPU/GPU coupled resonance micro-battery
- Battery D: optional NPU projection or priming probe if callable
- Battery E: persistence, hysteresis, and boundary perturbation battery
- Battery F: adversarial falsification against logging illusion, stale-path
  illusion, thermal confound, and residue contamination

## Hard Stops

- Stop if no callable heterogeneous comparison path can be defined at all.
- Stop if the environment cannot preserve enough state to compare windows.
- Stop if the only apparent effect is heat, timing, or output-directory drift.
- Stop if NPU remains inventory-only and GPU coupling adds no behavioral change.
- Stop if the phase starts narrating ontology instead of perturbation and
  response.
