# Phase 01.2.3.4.1.1.3.1.2.2 Research: Fresh-Eyes Analysis

## Core Question

What selects the high versus low same-family regime before row cpu_a starts?

## Fresh-Eyes Observations

### 1. Two Separate Axes of Variation

The prior packets show TWO independent axes, not one:

**Session-level duration axis:**
- Repaired packet: ALL rows ~155s regardless of value regime
- Confirmation replay: ALL rows ~200s
- Localization: ALL rows ~200s

**Row-level value axis:**
- HIGH regime: delta_E ~88-89, coherence ~0.77
- LOW regime: delta_E ~75-76, coherence ~0.877-0.893

The repaired packet was the ONLY "fast session" (155s). It was also the only one with
mostly HIGH values. All subsequent runs were "slow sessions" (200s).

### 2. Ambient CWD Is Littered With Residual Output Files

The current `/data/local/tmp` contains:
- 3 localization packet outputs: `f2_b_cpu_a.jsonl`, `f2_b_gpu_a.jsonl`, `f2_b_cpu_b.jsonl`
- 4 confirmation replay outputs: `f2_replay_cpu_a.jsonl` through `f2_replay_cpu_b.jsonl`
- 2 older run outputs: `f2_run_cpu_a.jsonl`, `f2_run_gpu_a.jsonl`
- 6+ chamber phase outputs: `phase_01_2_3_4_1_1_3_*`
- 2 codex probe outputs (zero-byte): `codex_phase_probe_*.jsonl`

The prior cleanup protocol only `rm -f` the SPECIFIC output file for the current row.
It never cleaned the other ambient .jsonl residuals.

### 3. PhonemePatterns_v1.bin Is ABSENT

The dm3_runner defaults to `--patterns PhonemePatterns_v1.bin` but this file does not
exist on the device. The binary's handling of this missing default is uncharacterized.
The STATE.md already notes this as an observation, but no prior phase tested whether
its presence or absence changes the regime.

### 4. The Repaired Packet Was 23 Hours Before Confirmation

- Repaired: 2026-04-15 ~00:00 UTC
- Confirmation: 2026-04-15 ~22:58 UTC
- Localization: 2026-04-15 ~23:13 UTC

The 23-hour gap allowed full device cooldown, filesystem cache eviction, CPU governor
state reset, and background system maintenance.  The 15-minute gap between confirmation
and localization preserved whatever state the confirmation left.

### 5. Hypothesis Candidates For The Regime Selector

Ranked by testability and prior evidence:

| # | Candidate | Mechanism | Testable by |
|---|-----------|-----------|-------------|
| H1 | Residual output files in CWD | Binary reads CWD listing or probes for prior outputs at startup | Deep clean ALL .jsonl before run vs shallow clean |
| H2 | Filesystem page cache state | Binary loads differently from cold flash vs warm cache | `sync && echo 3 > /proc/sys/vm/drop_caches` (needs root-equiv) or long idle |
| H3 | CPU frequency governor state | Scheduler assigns different core tier after idle vs after recent compute | Check `/sys/devices/system/cpu/cpu*/cpufreq/scaling_cur_freq` before/after |
| H4 | Internal binary mmap or tempfile residue | Binary leaves memory-mapped regions or temp files that survive between invocations | Check `/proc/PID/maps` during run, check for hidden temp files |
| H5 | Random initialization seed | Binary seeds from `/dev/urandom` or time-based, producing different trajectories | Multiple immediate reruns to see if regime is stable within a session |

### 6. Battery Design From These Observations

**Deep-clean cold anchor**: Remove ALL .jsonl and residual outputs from CWD, wait 120s,
capture full pre-state (battery, thermal, loadavg, process snapshot, binary hashes,
sidecar listing, cpu frequencies), then run cpu_a.

**Shallow-clean warm anchor**: After a retained run, remove ONLY the specific output file
(matching prior protocol), capture pre-state, immediately run cpu_a.

**No-clean immediate re-anchor**: After a retained run, do NOT remove even the output file,
use a different output filename, capture pre-state, immediately run cpu_a.

**Rapid repeat anchor**: Run cpu_a three times in immediate succession with different
output filenames and no cleanup between them. Test whether the regime is session-stable
or row-volatile.
