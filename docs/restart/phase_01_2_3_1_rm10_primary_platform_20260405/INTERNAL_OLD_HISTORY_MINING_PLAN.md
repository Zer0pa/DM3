# Internal Old History Mining Plan

## Purpose

This branch inherits two older RM10-side history families:

1. the governed SoC / Genesis line
2. the later DM3 prototype residue

They matter because they may contain executable clues, observable candidates,
role-partition hints, and failure signatures.
They do not matter as present authority.

This plan fixes what later history mining is allowed to extract from each.

## Branch Anchors

- Branch hypothesis:
  RM10-primary engineering and early heterogeneous execution may produce a more
  informative evidence surface than launcher-first continuation.
- Branch context:
  older histories must stay in a mining posture, not a recovery-worship
  posture.
- Mainline caution under test:
  the current branch must not erase the readiness ceiling by importing old
  labels as if they were present-tense facts.

## Source Families

| Family | Examples on the current device or repo surface | Trust class | What it is good for | What it is not allowed to do |
| --- | --- | --- | --- | --- |
| Governed SoC / Genesis line | `genesis_cli`, `snic_workspace_a83f`, governed replay ledgers, witness-floor docs | governed clue | command identity, receipt expectations, validator behavior, checkpoint discipline, CPU control candidates | proving new hybrid semantics |
| Later DM3 prototype residue | `dm3_runner`, bundled assets, preserved logs, hybrid strings, device-side residue paths | residue clue | names, assets, observable hints, boundary/interior metrics, accelerator-adjacent clues, failure signatures | proving a live current route or role without new evidence |
| Remembered or inferred folklore | architecture memories, unnamed launcher stories, retrospective interpretations | ambiguity at best | caution only if tied back to a real artifact later | entering any index as an actionable clue |

## Mining Outputs Required

Later plans must produce exactly these mining artifacts:

- `HISTORY_MINING_LEDGER.md`
- `HISTORY_CLUE_INDEX.md`
- `HISTORY_OBSERVABLES_INDEX.md`
- `HISTORY_ROLE_PARTITION_INDEX.md`
- `HISTORY_FAILURE_PATTERN_INDEX.md`

This plan defines what goes into each.

## Tagging Rules

Every mined item must carry all three tags:

1. `source_family`
   `governed_soc | dm3_residue | ambiguity`
2. `trust_class`
   `governed_clue | residue_clue | ambiguity`
3. `branch_action`
   `test_now | use_as_warning | hold_for_later | reject`

If one of these tags cannot be assigned, the item does not belong in the
actionable indexes.

## What To Mine From The Governed SoC / Genesis Line

Mine:

- explicit command surfaces
- receipt and hash fields
- validator expectations and failure modes
- checkpoint and resume identities
- known CPU-governed micro-battery shapes
- witness-floor comparators for later RM10-first runs

Do not mine:

- proof that the branch has already achieved device sovereignty
- proof that the current branch CPU control is already stable

## What To Mine From Later DM3 Residue

Mine:

- asset names and pairings
- observable names and metric labels
- boundary/interior split hints
- stage or role split hints touching CPU, GPU, DSP, NSP, or mixed execution
- wrapper, path, and staging clues only when backed by a real artifact
- failure signatures, dead ends, and ambiguous behaviors worth avoiding

Do not mine:

- task labels as if they were live current semantics
- accelerator presence as proof of user-space reachability
- preserved output headers as proof of branch-ready observables

## Mandatory Split Inside The Mining Work

Later history docs must keep these lines separate:

### Governed line

Use for:

- run discipline
- receipt discipline
- validator and baseline expectations
- CPU control design constraints

### Residue line

Use for:

- naming candidates
- observable candidates
- role-partition hypotheses
- failure warnings
- asset expectations

If a later document merges these into one undifferentiated "old history", it is
out of contract.

## Actionability Rules

A history item is actionable only if it improves one of these branch decisions:

- which common observable family to test first
- what exact assets or paths to inspect
- what role partition to try or refuse
- what failure mode should trigger a kill or abstain
- what receipt or checkpoint fields later plans must preserve

If a history item does not sharpen a future branch action, it is background
noise and should not be promoted.

## Explicit Rejection Rules

Reject any history item that depends on:

- same-binary bundled `G2` route hunting
- unmapped manifesto labels used as if they were branch truth
- remembered architecture with no surviving locator
- heat, performance, or aesthetic memory without an observable consequence

## Output Contract For Plan 02

Plan `02` must produce:

### Mining ledger

One custody-aware ledger that names:

- source family
- artifact location
- trust class
- why it matters to the branch

### Clue index

Executable clues only:

- paths
- asset expectations
- wrapper or launcher hints
- command-form expectations

### Observables index

Observable candidates only:

- what could be measured
- whether it is governed, residue-derived, or ambiguous
- what lane or battery family it might inform

### Role-partition index

Role hints only:

- CPU control roles
- GPU parity or assist roles
- NPU or DSP assist roles
- mixed-stage split hints

### Failure-pattern index

Failure warnings only:

- validation failures
- route collapse patterns
- thermal or staging hazards
- dead-end task families

## Hand-Off To Later Plans

Plan `03` will use the mining outputs to constrain runbooks.
Plan `04` will use them to choose battery families and stop rules.

That means the mining work should make later execution narrower, not broader.

## Bottom Line

The branch is allowed to learn from old RM10-side histories.
It is not allowed to inherit old confidence.
Anything mined from those histories must arrive in later plans as a clue,
warning, or ambiguity, never as proof.
