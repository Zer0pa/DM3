# Enterprise Drift Elimination And Engineering Completion PRD

Last refreshed: `2026-04-05`

## Mission

Turn the RM10-primary hypothesis branch into a clean, truthful, enterprise-grade
engineering surface that can support the next scientific brief without startup
drift, authority confusion, or inflated hardware claims.

This is not a proof campaign.

The object is still unknown.
Its possible utility is still unknown.
Its lawful observables are still under test.

The purpose of this programme is narrower and more disciplined:

- delete drift
- preserve authority
- harden the instrument
- expose incomplete engineering
- return a branch that can be trusted by the next operator

## Operating Posture

Keep this order fixed:

1. engineering truth
2. measurement and receipts
3. bounded interpretation

Do not invert it.

The Red Magic 10 Pro is the primary platform for this branch.
The Mac remains a support lane and the only current `source_built` authority
lane.

All object-level stories remain hypotheses.
Everything is falsifiable.
An abstain is acceptable.
A negative result is acceptable.
Uncontrolled drift is not acceptable.

## Current Branch Truth Floor

The PRD starts from these frozen facts:

- governed `F1` Genesis CPU control on RM10 is the only trustworthy branch-local
  scientific instrument
- the live governed `F1` bridge to an accelerator-bearing same-family surface is
  closed on the current `genesis_cli`
- bundled-residue `F2` is callable but remains `unstable_feasibility`
- NPU is inventory-only and `ABSTAIN`
- explicit heterogeneous role partition is still `ABSTAIN`
- fresh RM10 Genesis lanes are not source parity
- validator and canonical-target repair remain unfinished
- the branch still contains stale startup and operating surfaces that can mislead
  a fresh agent

## Primary Objective

Produce a branch state where a fresh high-reasoning cleanup or execution agent
can start from the repo itself, without oral context, and accurately determine:

- what is authority
- what is historical
- what is stale
- what engineering remains incomplete
- what the next bounded engineering move should be

## Non-Goals

Do not optimize for any of the following:

- proving what DM3 "is"
- widening `F2` residue feasibility into a bridge narrative
- narrating inventory-only NPU surfaces as a programme
- claiming explicit heterogeneous success without handoff artifacts
- promoting fresh RM10 executability into source-build parity
- preserving stale startup docs because they are familiar
- producing a polished pack that hides unresolved engineering

## Programme Questions

The cleanup and completion programme must answer:

1. Which documents currently outrank the truth by accident?
2. Which startup surfaces still point to the wrong repo, wrong phase, wrong
   build class, or wrong execution order?
3. Which engineering gaps remain before the next scientific brief is honest?
4. Which of those gaps are infrastructure gaps, provenance gaps, or evidence
   gaps?
5. What exact bounded next move most improves the branch without inflating any
   claim?

## Acceptance Gates

The programme is complete only when all of the following are true:

1. the startup prompt points to the correct repo, correct state, and correct
   reading order
2. active control docs no longer claim stale repo roots or stale phase
   sequencing
3. source-vs-prebuilt boundaries are explicit wherever execution surfaces are
   discussed
4. the engineering gap ledger is complete enough that the next operator does not
   need oral history
5. the branch can distinguish:
   - governed `F1`
   - residue `F2`
   - inventory-only NPU
   - abstained heterogeneous work
6. the next bounded engineering move is explicit and narrower than the current
   uncertainty

## Mandatory Sub-Agent Structure

The orchestrator must stay lean.
Use extra-high reasoning sub-agents for bounded ownership.

Minimum structure:

1. systems and platform
2. authority and drift
3. validator and provenance
4. accelerator boundary
5. logging and custody
6. falsifier
7. overclaim verifier
8. underclaim verifier
9. final pack writer

Each sub-agent must own a disjoint question or write scope.
The orchestrator integrates; it does not absorb all local detail.

## Commit And Logging Discipline

Every substantive cleanup or hardening action must leave:

- a factual commit message
- updated ledgers when drift is quarantined or deleted
- exact file references for changed startup surfaces

Do not hide cleanup inside omnibus prose commits.

## Stopping Rules

Stop and write a blocker note if:

- branch truth cannot be restored without changing evidence
- provenance and cleanup are in tension and you cannot safely separate them
- validator failure cannot be localized cleanly
- the only apparent progress comes from softer wording
- the proposed next move expands the claim surface instead of narrowing the
  engineering question

## Final Deliverables

The final package for this PRD must contain:

- updated startup surface
- updated operating map
- corrected branch hypothesis metadata
- drift and authority ledgers
- engineering gap ledger
- operations and receipt hardening note
- next-brief readiness memo

If any one of those is missing, the branch is not ready.
