## 3. New data since last brief (Session 7 close → now)

### Phase A.1 — μ replicate (closed)

N=10 independent invocations of `exp_k2_scars --steps 20` on SY_v1 tags + xnor_train:

```
r01..r10: best_uplift = 1.324074  (×10, bit-identical)
          max_scar_weight = 1.048401
          duration_sec: 259 — 1464 (wall-clock varies w/ thermal; output invariant)
```

Pre-registered PASS: all ten ≥ 0.05 AND median within ±30 % of 1.324. Both trivially satisfied because every run IS the median.

**What this tells us**: `exp_k2_scars` is **deterministic at fixed config**. No RNG, no noise, no seed matters. This is arguably stronger than the pre-registration asked for, and is a candidate claim (ξ).

### Phase A.2 — overfit boundary (closed)

21 runs, `--steps ∈ {20,25,30,35,40,45,50}`, N=3 each. Every N=3 set is bit-identical within itself.

| --steps | best_uplift | max_scar_weight |
|---|---|---|
| 20 | 1.324074 | 1.048401 |
| 25 | 1.380150 | 0.955608 |
| 30 | **1.644524** ← peak | 0.868061 |
| 35 | 1.405548 | 0.787437 |
| 40 | 1.642128 | 0.714148 |
| 45 | 1.332733 | 0.647856 |
| 50 | **0.000000** ← cliff | 0.588307 |

**Observations:**
- `max_scar_weight` decreases monotonically with `--steps` — more lessons produce smaller per-edge updates
- `best_uplift` is non-monotone but **always positive until it drops sharply to zero** between steps=45 and steps=50
- The overfit transition is a **cliff, not a slope** — candidate claim (ο)
- Peak uplift at steps=30 is 24 % higher than steps=20 (where μ was originally observed)

### Phase A.3 — cross-graph (9/12 landed)

4 cells (SY_v1, SY_v2, RA_v1, RA_v2) × N=3 at `--steps 20`:

| Cell | best_uplift (N=3 identical) | max_scar_weight |
|---|---|---|
| SY_v1 (default) | 1.324074 | 1.048401 |
| SY_v2 | 0.806000 | 0.965928 |
| RA_v1 | 1.341583 | 1.047126 |
| RA_v2 | *(pending — r1 in flight)* | |

**The headline**: tags (v1→v2) matter **a lot** for learning magnitude (−39 %); adj (SY→RA) matters **barely** (+1.3 %).

This is the **opposite structure** from the R1/R2 gate-flip finding: on the gate surface, R1 ↔ adj and R2 ↔ tags with approximately equal strength. On the learning surface, tags dominate and adj is nearly silent.

Waiting for RA_v2 to confirm (or break) the tags-dominance story:
- If `RA_v2 ≈ 0.8xx` → tags-dominance confirmed; adj really is silent
- If `RA_v2 ≈ 1.0–1.1` → interaction term exists
- If `RA_v2 ≈ 0` → something breaks under double-swap

### Phase A.4 — cross-dataset (pending)

3 datasets (xnor_train / xnor_mini / xnor_test) × N=3. Queued to run after A.3 closes. Tests whether μ generalizes across dataset content.

### Engineering-side finding (also new)

The Qualcomm PMIC sensor `pmih010x_lite_tz` has ~hours of thermal inertia. In the first Phase A run it saturated at 71–72 °C and stayed there for 2+ hours while CPU cores were in the low 30s. The old `run_cell.sh` thermal gate read "worst across all zones" and walled 14 of 21 A.2 runs + all 12 A.3 + all 9 A.4 during that window.

**Fix**: gate now filters to `cpu-*/cpuss-*/gpuss-*` sensor types only. PMIC, DDR, NSP, pm-* excluded. Verified on resume: CPU-worst = 32 °C while all-sensor-worst = 62 °C — the PMIC reading still lagging.

**Engineering IS NOT finding**: the phone's thermal budget is **not** what the default all-sensor gate measured. The PMIC saturates under compute but does not indicate an actual thermal emergency on the compute side.

## 6. Decision request — do we rejig?

Three questions for the team. Answer any or all.

### Q1. Does Phase A already tell us enough to promote ξ/ο/π?

A.3 will be done in ~90 min; A.4 in another ~2 h. If those close cleanly, we have **three Phase A claims** plus a thermal-gate engineering IS NOT, all landable before Session 8 end. Is that sufficient for the Phase F ledger, or do we want more pressure before promoting?

**Default plan**: close A.4, write Phase A final section, continue into Phase B.
**Alternative**: Phase A looks strong enough — cut B.1/B.2 scope, push straight to Phase C (gate-surface structure) because the R3-flip hunt is where the biggest open scientific question lives.

### Q2. Phase B scope — all three sub-phases, or only the CLI-audit one?

B.1 (LEARNS probes on i1, i2, h1_h2 @ N=5) is a big spend if those tasks turn out to be CLI-decorative like truth-sensor was (κ) and resonance_r3 was (ν). B.3 (systematic flag-responsiveness audit across 12 tasks × all flags) gives us the "IS NOT tunable via X" side of the ledger **as a table**, which is more durable than individual claim receipts.

**Default plan**: run all three sub-phases.
**Alternative**: do B.3 first; if it shows multiple tasks are CLI-decorative, skip B.1 on those tasks and save ~10 h of compute.

### Q3. Phase E — do we want Mode A at all?

Phase E requires building a Wikipedia corpus subset, running sentence embeddings, building a region-mapping function from embeddings into the 380-vertex ontology, and evaluating retrieval accuracy. **That infrastructure build is outside the phone** — it's host-side work. It was in scope as "the biggest open scientific question" (can DM3 diagnose external labeled content) but it's also the phase with the most external infrastructure risk.

**Default plan**: build E.0 infra between sessions, run E.1/E.2 in Session 9.
**Alternative**: skip E and let Phase F's OPEN_QUESTIONS ledger record "Mode A retrieval" as a question for the future. This keeps Session 8 pure-binary-characterization.

### Q4. Anything else we're missing?

Is there a cell we should add now while Phase A is live? Some candidates we haven't discussed:
- **Noise-schedule sweep on exp_k2_scars** — right now `--noise` cycles {0.1, 0.2} inside the task, fixed. What if a user's data had lower or higher noise? (Not obvious this is runnable without binary mod.)
- **Dataset-size sweep** — we run xnor_train by default; does learning scale with dataset size?
- **Multi-task sequential training** — can `exp_k2_scars` on dataset A followed by dataset B retain A?
- **Cross-graph replication of the overfit cliff** — A.2 was SY_v1 only; if the cliff moves with graph or tags, that's a different kind of claim.
