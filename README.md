# DM3

## What This Is
DM3 is a compiled deterministic graph-theoretic computation over a
fixed 380-vertex C3-symmetric graph constructed from exact-rational
2D Sri Yantra geometry and lifted into 3D by toroidal twist. The
current public object is a precompiled Rust binary, `dm3_runner`
`sha256=daaaa84a052b60523bf9d63152f1154225abf119c279aa4b3aabf14487279672`,
running a 380×192 = 72,960-dimensional relaxation system with twelve
callable task modes. Session 8 now promotes fixed-config
`exp_k2_scars` determinism, the sharp `49→50` overfit cliff, and ARM64
cross-platform determinism for the tested baseline values. It also
carries scoped candidates for dataset invariance, `RA+v2+steps=20`
zero learning, a trimodal sawtooth K2 curve, and three
`--steps`-decorative task surfaces. DM3 is instrument-shaped rather than
codec-shaped, so the parser-required identity rows below are used as
product tags for this diagnostic surface.

## Reproducibility Envelope
DM3's strongest current public result is deterministic receipt matching:
same Android aarch64 binary, same input files, same emitted KPI values,
different ARM64 silicon. The τ packet matches all 10 tested
`exp_k2_scars` floats across Snapdragon 8 Elite native Android
(Red Magic 10 Pro) and Apple M1 Android ARM64 AVD execution under
Hypervisor.framework. This removes the current "only this phone" reading
for the tested ARM64 baseline surface.

The current repo package checks the public surface and proof anchors via
`zpe-dm3`. A one-command public rerun wrapper named `dm3-verify` is not
present in this checkout; external verification is presently the
receipt path documented in `CLAIMS.md`, the τ packet, and the artifact
trees named below.

For deterministic task surfaces, the public standard is exact match or
exact diff: binary/input hashes, emitted KPI values, and retained
receipt files. The harmonic basin line is explicitly different because
it is stochastic in selection and is therefore governed by the promoted
Wilson-confidence interval evidence, not a bit-identical output claim.

## Receipt Discipline
Claims are not promoted from plausible-looking curves. The public ledger
uses `CANDIDATE`, `CONFIRMED`, `WEAKENED`, `RETRACTED`, and
`REJECTED-BEFORE-PROMOTION` states with explicit kill criteria. Session
8 is the current example: the coarse `σ` wording and the interim `σ′`
wording were killed by finer sampling before promotion; the active
curve claim is only candidate `σ″`. Nulls and decorative-control
findings are kept on the same surface as positive results.

## What We Prove
- DM3 is a compiled structural diagnostic on a fixed 380-vertex C3-symmetric graph with a 72,960-dimensional relaxation state.
- The `harmonic` task exhibits reproducible HIGH and LOW basins, with IID default basin selection and overlapping Wilson confidence intervals across the six promoted Session 7 dynamics arms `(665 total episodes)`.
- The Session 7 smoke receipt fixes a default `exp_r1_r4_campaign` reproducibility fingerprint at canonical SHA `9006df4ec02c8872...` on the `--steps 5` surface.
- The 2×2 gate cross-control is closed: `R1` is pure `--adj`, `R2` is pure `--tags`, and `RA+v2+steps=50` flips both together while raising `r4.transfer_ratio` from `1.369` to `2.678`.
- `exp_k2_scars` is the first receipted positive learning line in the project: `best_uplift` rises monotonically through `--steps 20` and overfits at `50`.
- Session 8 strengthens fixed-config `exp_k2_scars` KPI determinism into the bedrock reproducibility line: the engineer final report summarizes roughly 142 receipts across cells, days, and two ARM64 execution surfaces, while this repo keeps the original Phase A count seam visible.
- Session 8 sharpens the overfit cliff to `--steps 49` (`1.819397`) → `--steps 50` (`0.000000`).
- The τ lane confirms ARM64 cross-platform determinism for the tested `exp_k2_scars` baseline values: RM10 native Android and Apple M1 Android ARM64 emulator match on all 10 emitted KPI floats.
- Session 8 candidates are scoped: xnor dataset invariance at the μ baseline, `RA+v2+steps=20` zero learning, a trimodal sawtooth K2 curve, and three `--steps`-decorative task surfaces at `steps 1` versus `20`.
- `exp_k3_truth_sensor` and `resonance_r3` both do real internal work, but their decorative-control claims are narrow: sensor strength/threshold for K3, and `--steps` for `resonance_r3`.

## What We Don't Claim
- DM3 is not a portfolio-wide platform component; it stands alone as its own product.
- DM3 is not an AI system, and transformer-created bistability is not a live claim.
- Rotational angle `60°` does not uniquely break C3; Claim gamma remains retracted.
- The positive learning claim is narrow and task-specific to `exp_k2_scars`; this root surface does not promote a broad “DM3 learns” statement.
- `exp_k3_truth_sensor` and `resonance_r3` are not promoted as tunable CLI surfaces.
- Claim `σ″` is not CONFIRMED yet; the trimodal sawtooth line is a scoped baseline-surface candidate pending deeper shape and transfer tests.
- Earlier `σ` and `σ′` K2 curve wordings are not active claims; they were rejected-before-promoted by finer A5/A6 receipts.
- Claim `ρ` is not a broad RA+v2 ceiling; it is scoped to `RA+v2+steps=20`.
- Claim `φ` is not a broad “these tasks never respond” claim; it is scoped to `--steps 1` versus `20` under tested defaults.
- Claim `κ` is not a universal `--steps`-decorative claim for `exp_k3_truth_sensor`; B3 shows `--steps` changes absolute baseline and gap values.
- This repo does not claim source-level reproducibility for the current hybrid `dm3_runner`; the source for this binary is not present in the known local repo/filesystem surface as of 2026-04-24.
- x86 cross-platform parity is not claimed; it is source-blocked until the current hybrid binary's source is recovered or an equivalent source-built lane exists.
- Mode A external-content diagnosis is not claimed or teased on this front door; the scaffold has not produced a promoted result.
- A packaged public `dm3-verify` command is not claimed in this checkout.
- The Session 7 final report closes a bit-level gate-layer substrate-null battery, but the standalone `S2_pinned / S4_airplane / S6_core` directory names cited in the handover are not exposed verbatim in this checkout; the repo keeps that mismatch visible instead of smoothing it over.
- The Session 8 final report says Phase A closed with 55 total-run accounting, while the local mirror contains 52 per-run receipt/log pairs; the count seam is explicit in `repo_stage/REPO_AGENT_FINDINGS.md`.
- The Session 8 airplane-mode deviation affecting 40 reported receipts is not hidden.
- No external competitive benchmark pack is promoted on this root surface yet.
